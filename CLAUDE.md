## Project Overview

**obsidian-external-bridge** is an Obsidian community plugin (desktop only) that creates lightweight placeholder notes in your vault for files stored in external folders. This solves the problem of Obsidian Sync size limits — large external files (PDFs, videos, archives) are represented by small YAML-rich notes instead of syncing the actual files.

- **Plugin ID:** `external-bridge`
- **Min Obsidian version:** 1.4.0
- **Platform:** Desktop only (requires Node.js `fs` module via Electron)
- **Current version:** 1.1.0 (see `manifest.json` and `versions.json`)
- **Author:** bramzilla

## Architecture: Single-File Plugin

The entire plugin lives in `src/main.ts`. There is **no module splitting**. All classes, interfaces, and helpers are in one file. Keep it that way — avoid splitting into multiple files unless the user explicitly requests it.

### Key Interfaces (`src/main.ts`)

| Interface | Purpose |
|---|---|
| `BridgedFolder` | Configuration for one external-folder→vault mapping (id, paths, extensions, watch, customProperties) |
| `SyncLogEntry` | Record of a sync operation with counts and affected filenames |
| `ExternalBridgeSettings` | Plugin-wide settings (bridges array, defaults, debounce, auto-sync flags) |
| `BridgeHealth` | Union type: `'ok' \| 'unreachable' \| 'unknown'` |

### Key Classes

| Class | Responsibility |
|---|---|
| `FolderWatcher` | Wraps `fs.watch` with debouncing and polling fallback (5s interval) |
| `ExternalBridgePlugin` | Main plugin class — lifecycle, settings, watcher management, sync logic |
| `BridgeManagerModal` | Primary UI modal — lists bridges with health/watcher status, per-bridge actions |
| `AddBridgeModal` | Form modal for creating a new bridge |
| `EditBridgeModal` | Form modal for editing an existing bridge |
| `SyncLogModal` | Displays per-bridge sync history with expandable detail entries |
| `ExternalBridgeSettingTab` | Settings tab in Obsidian Settings → Community Plugins |

### Key Helper Functions

| Function | Purpose |
|---|---|
| `buildFrontmatter()` | Assembles YAML frontmatter block for a placeholder |
| `buildMetaBody()` | Generates the auto-managed content body (file info table, type-specific callout) |
| `buildFullPlaceholder()` | Combines frontmatter + meta body + user-content zone sentinels |
| `mergeUpdatedContent()` | Re-syncs a placeholder while preserving text between `<!-- bridge:user-content:start/end -->` |
| `escapeYamlValue()` | YAML-safe string escaping |
| `getFilesRecursive()` | Recursively list files filtered by extension list |
| `checkBridgeHealth()` | Checks if an external folder path is accessible |
| `pickFolder()` | Opens native folder picker dialog via Electron |
| `formatFileSize()` | Converts bytes to human-readable string (B, KB, MB, GB) |
| `generateId()` | Generates random 8-character bridge ID |

## Development Workflow

### Prerequisites

- Node.js (LTS)
- npm

### Setup

```bash
npm install
```

### Commands

```bash
npm run dev       # Watch mode — rebuilds main.js on file changes (inline sourcemaps)
npm run build     # Type-check + production build (no sourcemap, tree-shaken)
npm run version   # Bump version in manifest.json + versions.json, stage both
```

### Build System

- **Bundler:** esbuild (`esbuild.config.mjs`)
- **Entry:** `src/main.ts` → **Output:** `main.js`
- **Format:** CommonJS (required by Obsidian)
- **Target:** ES2018
- **Externals:** `obsidian`, `electron`, all Node.js builtins, CodeMirror/Lezer packages

In development mode (`npm run dev`), esbuild watches for changes and rebuilds with inline sourcemaps.

In production mode (`npm run build`), TypeScript is type-checked first (`tsc -noEmit`), then esbuild produces a minified bundle.

### Testing

There is **no automated test suite**. Testing is done manually by loading the plugin in an Obsidian vault. To test:

1. Run `npm run dev`
2. Copy or symlink `main.js`, `manifest.json`, and `styles.css` into `.obsidian/plugins/external-bridge/` in a test vault
3. Enable the plugin in Obsidian → Settings → Community Plugins

## Code Conventions

### TypeScript

- Strict mode is enabled (`tsconfig.json`)
- Target: ES6, module: ESNext
- All plugin classes extend Obsidian base classes (`Plugin`, `Modal`, `PluginSettingTab`)
- Avoid adding type annotations or docstrings to code you haven't modified
- Keep all code in `src/main.ts` — do not split into modules

### Sentinel-Based Content Preservation

Placeholder notes use HTML comment sentinels to distinguish auto-generated from user-written content:

```
<!-- bridge:meta:start -->
(auto-generated content — overwritten on every sync)
<!-- bridge:meta:end -->

<!-- bridge:user-content:start -->
(user notes — preserved across syncs)
<!-- bridge:user-content:end -->
```

The `mergeUpdatedContent()` function handles updating the meta section while preserving the user-content section. Never break this pattern.

### YAML Frontmatter

Placeholders include standard fields (`tags`, `external-path`, `external-mtime-ms`, `file-size`, `file-type`, `bridge-id`, `bridge-label`) plus any per-bridge `customProperties`. The `external-mtime-ms` field enables smart sync-skipping — if the file hasn't changed, the placeholder is not rewritten.

### File Watcher

`FolderWatcher` uses `fs.watch` as the primary mechanism with a polling fallback. The debounce interval is user-configurable (default: 1000ms). Always stop watchers via `stopAllWatchers()` in `onunload()` to prevent resource leaks.

### CSS

All styles are in `styles.css`. CSS variables are used for theming to support Obsidian's dark/light modes. Component styles follow the naming pattern `.bridge-*` (e.g., `.bridge-card`, `.bridge-health-dot`).

### Electron API Usage

Folder picking uses Electron's `remote` module with a fallback:
```ts
// Tries @electron/remote first, falls back to electron.remote
```
Do not use Electron APIs that are unavailable in Obsidian's Electron version or that require main-process access beyond what's demonstrated in `pickFolder()`.

## Obsidian Plugin API Conventions

- Access vault files through `this.app.vault` (never raw `fs` for vault files)
- Use `fs` (Node.js) only for **external** paths outside the vault
- Register commands via `this.addCommand()` in `onload()`
- Register ribbon icons via `this.addRibbonIcon()` in `onload()`
- Register settings tab via `this.addSettingTab()` in `onload()`
- Defer heavy startup work to `this.app.workspace.onLayoutReady()` callback
- Clean up all resources (watchers, listeners) in `onunload()`

---

## Settings & Persistence

Settings are persisted via Obsidian's `loadData()` / `saveData()` API (stored in `.obsidian/plugins/external-bridge/data.json`). The settings object is `ExternalBridgeSettings`. Always call `await this.saveSettings()` after mutating settings.

Default settings are defined in `ExternalBridgePlugin.loadSettings()` using `Object.assign({}, DEFAULT_SETTINGS, await this.loadData())`.

## Adding a New Feature — Checklist

1. **Types/interfaces** — add new fields to the relevant interface (`BridgedFolder`, `ExternalBridgeSettings`, etc.)
2. **Defaults** — update `DEFAULT_SETTINGS` if the feature adds a new setting
3. **Core logic** — implement in `ExternalBridgePlugin` (or a new helper function at module level)
4. **UI** — add controls in the appropriate modal (`AddBridgeModal`, `EditBridgeModal`, `ExternalBridgeSettingTab`)
5. **CSS** — add styles to `styles.css` following the `.bridge-*` naming convention
6. **Documentation** — update `FEATURES.md` and `README.md` if user-visible

## Version Bumping

To release a new version:

```bash
npm run version   # Prompts for new version, updates manifest.json + versions.json
git add manifest.json versions.json
git commit -m "chore: bump version to X.Y.Z"
```

Both `manifest.json` and `versions.json` must be updated together. `versions.json` maps each plugin version to the minimum required Obsidian version.
